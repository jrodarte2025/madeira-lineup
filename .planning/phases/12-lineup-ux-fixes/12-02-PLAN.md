<objective>
Close LUX-03: tap-to-sub is symmetric. Today, tapping a bench player then tapping a field slot moves them onto the field (works). Tapping a field player then tapping the bench does nothing today — users must double-tap (mobile) or double-click (desktop) to remove. This plan makes the bench a valid tap-target for field-selected players on both desktop and mobile.

Purpose: Subbing a player off the field is the most frequent game-day action. On the sideline, double-tap is finicky and the bench is the obvious "drop zone" for a selected field player. Make it work the way a coach's muscle memory expects.

Output: Tap field player -> tap bench anywhere -> that player goes to bench. Position becomes empty. No other changes to existing flows.
</objective>

  <action>
    Create a small helper function near the other handlers (around `handlePositionClick`, ~line 812):

    ```javascript
    const handleBenchTap = (e) => {
      // Only trigger if the tap lands on the bench BACKGROUND, not on a chip.
      // Chips stopPropagation via their own onClick or we check e.target === e.currentTarget.
      if (e.target !== e.currentTarget) return;
      if (!selectedPlayer) return;
      const selectedPosIndex = lineup.indexOf(selectedPlayer);
      if (selectedPosIndex === -1) return; // selected player is already on bench — no-op
      removeFromPosition(selectedPosIndex);
      setSelectedPlayer(null);
    };
    ```

    Wire it on both bench renders:

    1. **Desktop bench bar** (~line 1312). Add `onClick={handleBenchTap}` to the outer div that already has `onDragOver={handleRosterDragOver}`. Also add a `style.cursor: selectedPlayer && lineup.indexOf(selectedPlayer) !== -1 ? 'pointer' : 'default'` so the affordance is visible when a field player is selected.

    2. **Mobile chipstrip** (~line 1129). Add `onClick={handleBenchTap}` to the chipstrip div (the one with `data-drop-id="chipstrip"`). The existing chips each have their own onClick/onTouchEnd that selects that bench player — those MUST continue to work. The `e.target !== e.currentTarget` guard prevents a chip click from bubbling into a sub-off. Verify by reading each chip's handler (`onClick={(e) => { if ('ontouchstart' in window) return; handlePlayerClick(p.id); }}`) — chip onClick does NOT call stopPropagation, so bubbling WILL happen without the guard. Keep the guard.

    3. **Desktop sidebar roster panel** (~line 1100, `<div onDragOver={handleRosterDragOver} onDrop={handleRosterDrop}...>`). Add `onClick={handleBenchTap}` here too. The inner `RosterContent` renders `PlayerChip`s that already have their own onClick handlers; the guard `e.target !== e.currentTarget` is strict — but `PlayerChip`'s root div will bubble events up. Verify bubbling doesn't cause a false sub-off by testing manually (Task 3).
